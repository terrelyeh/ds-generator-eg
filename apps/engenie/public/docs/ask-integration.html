EnGenie Ask — 給每個部門的
AI 產品知識助理

把 EnGenius 的產品規格、文件、FAQ、上傳資料變成一個能對話的 AI 助理，讓你的團隊或客戶用自然語言直接問答——並用最適合你的形式整合進現有網站或工具。

🟦 浮動 Widget 🔗 整頁入口 ⚙️ Search API 🔒 各自 scope / key / 角色

1這是什麼

每個部門可以擁有自己的 Ask 入口（/ask/<你的代號>），背後共用同一個 EnGenius 知識庫，但各自設定：

  • 知識範圍 (scope)：限定到某 Solution / 產品線 / 型號 / 來源類型，只回答你關心的範圍。
  • 生成模式：用系統共用的 AI key，或帶你自己的 key（見第 3 節）。
  • 角色與歡迎語：預設 persona／對象、歡迎標題、範例問題，可鎖定或開放使用者切換。
  • 存取控制：可設 passcode（半私密）或開放（無摩擦對外）。
知識庫怎麼來？ 由 SpecHub 集中維護：產品規格、Gitbook 文件、Help Center、Google Docs、網頁、WiFi 法規，加上你可以自己新增的文字片段與上傳 PDF（PDF 由 AI 讀取，含表格、圖片描述、掃描檔 OCR）。

同一個 workspace、同一套知識，三種對外形式任你挑、也可並用：

直接分享 /ask/<代號> 網址（手機也能像 App 一樣加到主畫面）。不用改任何網站。

⚙️

Search API

給：自建 App / AI agent

機器對機器的 JSON 檢索 API（/api/v1/search，Bearer key、可限 scope、有流量限制）。把知識庫接進你自己的產品——消費端可以是傳統程式，也可以是 AI agent（Claude Code skill / MCP）。

RESTful · MCP-ready
前兩種（Widget／整頁）是對人的完整聊天體驗；第三種（Search API）是對程式的檢索結果。Search API 的完整規格見 RAG Search API 文件。

換個角度：其實是「兩個家族」——誰來當回答的大腦

上面三種，差別其實在「生成答案的 LLM 在誰那邊」，可收斂成兩家：

家族回答的「大腦」對應形式
A · EnGenie 自己答EnGenie 的 RAG 管線（檢索＋生成）浮動 Widget、整頁入口（對人、完整聊天）
B · EnGenie 只給料，別人的腦答呼叫方Search API → 消費端可是 傳統 App，也可是 AI agent
B 家族最進階的一種：把知識接進 AI agent。做成一個 Claude Code skill（現有的 engenius-kb）或 MCP server（設計草案見 EnGenie Knowledge MCP），EnGenie 知識就變成使用者 AI 工具裡的「一個工具」——直接在他本來的工作流（Claude Code／Cursor…）裡問，還能跟他的其他工具一起用。架構觀念見 Agent 架構與多 Agent 架構文件。

3三種 LLM 模式

「AI 要用誰的 key、誰付費」可以三選一：

模式API key 從哪來計費最適合
Shared key 系統共用的 AI key（你不用準備） 由 SpecHub 統一 大多數部門；最省事
Workspace BYOK 管理員幫整個 workspace 設一把你的 key 記在你部門的帳上 要用自己帳號 / 集中計費的團隊
User BYOK 每位使用者第一次用時自己貼自己的 key（存在他的瀏覽器） 各自負擔 每個人用自己額度的情境
對外 Widget 建議用 Shared 或 Workspace BYOK。 User BYOK 會要求每位訪客先貼自己的 key，不適合公開的浮動 widget。

4知識範圍與權限

知識範圍 (scope)

每個 workspace 可限定回答範圍：Solution → 產品線 → 型號 三層，加上來源類型（產品規格 / 文件 / Help Center / 網頁 / 文字片段 / PDF…）。留空＝全部。例如「只回答 Cloud AP 產品線的規格與文件」。

存取控制

  • 有 passcode：訪客輸入一次通行碼（半私密，例如內部工具）。
  • 無 passcode：打開即用，最適合對外、客戶面的 widget。
  • 流量保護：每分鐘 / 每日上限可調，保護共用資源。
  • 鎖定設定：可關閉「使用者自行切換模型 / 角色」，固定成你設定的樣子。

5三步驟開始

  1. 告訴我們需求：你的部門代號、知識範圍、要不要 passcode、用哪種 LLM 模式。管理員在 Settings → Ask Workspaces 幫你建立。
  2. 拿到你的入口：得到一條 /ask/<代號> 連結，以及一段 widget snippet（管理頁「Embed」一鍵複製）。
  3. 整合 / 分享：把 snippet 貼進你的網站、或直接分享連結、或用 API key 串進你的 App。完成。

6複製即用

① 浮動 Widget（貼在你網站 </body> 前）

<script
  src="https://engenie-eg.vercel.app/widget.js"
  data-workspace="你的代號"
  data-title="你的助理名稱"
  async></script>

可選參數：data-color="#03a9f4"（按鈕顏色）、data-position="left"（改成左下角）。

② 整頁入口（直接分享連結）

https://engenie-eg.vercel.app/ask/你的代號

實際範例（demo workspace）：開啟 /ask/demo-widget ↗

③ Search API（程式串接）

curl -X POST https://engenie-eg.vercel.app/api/v1/search \
  -H "Authorization: Bearer sk_live_你的key" \
  -H "Content-Type: application/json" \
  -d '{"query":"哪些 AP 支援 WiFi 7?","top_k":5}'

完整參數、scope、限流、錯誤碼見 RAG Search API 文件。

7常見問題

Widget 會影響我網站的樣式嗎？

不會。聊天介面跑在獨立的 iframe 裡，你的 CSS 不會干擾它、它也不會干擾你的頁面。

需要登入 SpecHub 嗎？

不需要。Widget／整頁入口給的是你部門專屬連結，訪客（含外部客戶）不需要 SpecHub 帳號；若有設 passcode 則輸入一次即可。

資料安全嗎？跨網站能用嗎？

可以。Widget 用安全 token（非 cookie）認證，在 Safari / Chrome 等封鎖第三方 cookie 的瀏覽器都正常運作。User BYOK 模式下，使用者的 key 只存在他自己的瀏覽器、不會存到伺服器。

知識可以只限我們部門的內容嗎？

可以。透過 scope 限定 Solution／產品線／型號／來源類型；也能上傳你部門自己的 PDF 與文字片段。

回答的語言？

跟著提問語言走——中文問中文答、英文問英文答、日文問日文答。

對話歷史存在哪裡？保留多久？

對話歷史只存在使用者自己的瀏覽器（localStorage），不會上傳伺服器、也不與他人共享。重點規則：

  • 每個入口（workspace）各自獨立，彼此看不到對方的歷史。
  • 最多保留最近約 30 筆對話（超過自動汰舊）。
  • 換裝置、換瀏覽器、清除瀏覽資料、或無痕視窗全部關閉後就會消失。
  • 嵌入式 widget 的歷史是依「所在網站」分開存的——同一個入口放在不同網站，歷史不互通。
  • 因為存在前端，不適合當長期/正式紀錄；需要稽核或跨裝置保存請另循正式管道。
