Ask SpecHub — RAG 系統說明

Retrieval-Augmented Generation for EnGenius Product Specifications  |  Last updated: 2026-04-12

1 功能概述

Ask SpecHub 是 Product SpecHub 內建的 AI 產品查詢助手。同事可以用自然語言（中文或英文）詢問任何 EnGenius 產品規格相關的問題，系統會從已索引的產品資料中找到最相關的內容，交由 AI 產生精準回答並附上來源。

核心能力

  • 規格查詢 — 「ESG510 的 VPN throughput 是多少？」
  • 產品比較 — 「ECC100 和 ECC500 差在哪裡？」
  • 條件篩選 — 「哪些 AP 支援 WiFi 7？」「Which switches support PoE++?」
  • 推薦建議 — 「推薦一台適合戶外的攝影機」
  • 多語言 — 支援中文和英文提問，AI 會用同語言回答

技術名詞

名詞說明
RAGRetrieval-Augmented Generation — 先搜尋相關資料，再交由 AI 生成回答。避免 AI 「幻覺」（憑空編造）
Embedding將文字轉換成數學向量（1536 維數字陣列），語意相近的文字在向量空間中距離也近
向量搜尋用 cosine similarity 計算問題向量和文件向量的相似度，找出最相關的文件
Chunk將文件拆成小片段。每個 chunk 獨立做 embedding，搜尋時回傳最相關的 chunks
pgvectorPostgreSQL 的向量搜尋擴充，直接在 Supabase 內執行，不需額外的向量資料庫

3 資料庫設計

documents 表 Schema

使用單一 documents 表儲存所有來源的文件 chunks 和向量，透過 source_type 區分不同來源。

CREATE TABLE public.documents (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- 來源識別
  source_type     TEXT NOT NULL,     -- 'product_spec' | 'gitbook' | 'web' | ...
  source_id       TEXT NOT NULL,     -- 產品 model_name / URL / Doc ID
  source_url      TEXT,              -- 原始連結（回答附引用用）

  -- 內容
  title           TEXT NOT NULL,     -- 顯示用標題
  chunk_index     INT NOT NULL DEFAULT 0,  -- 同文件第幾段
  content         TEXT NOT NULL,     -- 純文字內容
  token_count     INT,               -- 估算 token 數

  -- 彈性 metadata
  metadata        JSONB NOT NULL DEFAULT '{}',
  -- 例如: {"product_line": "Cloud Camera", "status": "active", "chunk_type": "overview"}

  -- 向量
  embedding       VECTOR(1536),      -- OpenAI text-embedding-3-small

  -- 變更偵測
  content_hash    TEXT,              -- SHA-256，內容不變就不重新 embed
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

索引名稱欄位用途

match_documents 函數

Supabase RPC 函數，執行向量相似度搜尋並支援篩選：

SELECT * FROM match_documents(
  query_embedding  := '[0.012, -0.034, ...]',  -- 問題的向量
  match_count      := 8,                         -- 回傳前 8 筆
  match_threshold  := 0.3,                       -- 相似度門檻
  filter_source_type := 'product_spec',         -- 只搜產品規格
  filter_metadata  := '{"product_line":"Cloud Camera"}'  -- 只搜特定產品線
);

4 AI 如何參與

整個 RAG 系統中有 兩次 AI 呼叫，各司其職：

Step 1: Embedding（文字 → 向量）

項目說明
用途將文字轉成 1536 維向量，捕捉語意（而非關鍵字）
模型OpenAI text-embedding-3-small
費用$0.02 / 1M tokens（極便宜，66 個產品 ≈ $0.001）
何時呼叫1) 建立/更新索引時（每個 chunk 一次）
2) 使用者提問時（每次提問一次）
為什麼不用關鍵字搜尋向量搜尋理解語意：「戶外攝影機」能匹配到 "IP67 outdoor dome camera"

Step 2: 向量搜尋（找相關文件）

項目說明
引擎Supabase pgvector（PostgreSQL 內建）
演算法IVFFlat index + cosine distance
回傳數量Top 8 最相關的 chunks
相似度門檻0.3（低於此值的結果不回傳）
費用零（SQL 查詢，不呼叫外部 API）

Step 3: LLM 回答（生成回答）

項目說明
用途基於搜尋到的文件，用自然語言回答問題
可選模型Claude Sonnet（預設）、GPT-4o、Gemini 2.5 Flash
System Prompt指定為 EnGenius 產品專家，只基於提供的 context 回答
語言偵測Prompt 指示 AI 用提問的語言回答（中文問中文答）
防幻覺明確要求「If the context doesn't contain enough information, say so」
為什麼需要兩步（搜尋 + LLM）而不是直接問 AI？ 直接問 ChatGPT「ECC100 的解析度是多少？」它可能會編一個答案（幻覺）。RAG 的做法是先從你自己的資料庫找到真實資料，再讓 AI 基於這些資料回答。AI 看到的是你提供的 context，不是它的訓練資料。

5 Chunking 策略

每個產品生成 2 個 chunks：

Chunk 0: Overview（產品概述）

捕捉「這個產品是什麼、能做什麼」 
ECC100 — Cloud-Managed AI Outdoor Dome Camera
Product Line: Cloud Camera (Cloud)
Status: active
Subtitle: Cloud-Managed AI Outdoor Dome Camera

Overview:
The ECC100 is a cloud-managed 2MP outdoor dome camera
with AI-powered detection and 256GB built-in storage...

Key Features:
• AI-powered people and vehicle detection
• 256GB built-in storage
• IP67 weatherproof rating
• ...

Chunk 1: Specifications（技術規格）

捕捉「具體的技術數值」 
ECC100 — Technical Specifications

[Camera]
  Image Sensor: 1/2.8" 2MP Progressive Scan CMOS
  Resolution: 1920 x 1080 (1080p)
  Lens: 2.8mm fixed

[Network]
  Protocols: TCP/IP, HTTPS, DHCP, DNS
  PoE: IEEE 802.3af

[Physical]
  Dimensions: 122 x 122 x 97 mm
  Weight: 420g
  ...
為什麼拆成兩個 chunk？ Overview 和 Specs 的語意差異很大。「推薦一台戶外攝影機」應該匹配 overview（因為 overview 提到了使用場景）；「ECC100 的 PoE 標準是什麼？」應該匹配 specs。拆開後搜尋更精準。

為什麼不需要更複雜的 chunking？

常見的 RAG 系統要處理長文件（PDF 報告、法律合約），需要複雜的 chunking（overlapping windows、recursive splitting）。但 EnGenius 產品規格有兩個特性：

  • 資料量小 — 每個產品 overview 約 200 tokens，specs 約 500-1000 tokens，遠小於 embedding 模型的 8191 token 上限
  • 結構清晰 — 每個產品就是一個獨立單位，不需要跨產品的 chunk

7 設定與使用

首次設定

1. 確認 API Key

RAG 系統需要兩組 API Key：

Key用途必要性
OpenAI API KeyEmbedding（text-embedding-3-small）必要 — 沒有就無法建索引和搜尋
Claude / GPT-4o / Gemini KeyLLM 回答生成至少一個 — 用於生成回答

到 /settings/api-keys 設定，或在 Vercel 環境變數中設定 OPENAI_API_KEY。

2. 執行 Migration

如果使用 Supabase MCP 或 CLI，migration 00009_add_rag_documents.sql 已經準備好。確認 pgvector extension 已啟用且 documents 表已建立。

3. 建立索引

  1. 進入 /ask 頁面
  2. 點擊 Index Products 按鈕
  3. 等待完成（66 個產品約 10-20 秒）
  4. 狀態列會顯示「X products indexed (Y chunks)」

日常使用

  1. 從 Navbar 點 Ask 進入
  2. 在輸入框打字或點擊範例問題
  3. 右下角可切換 AI provider（Claude / GPT-4o / Gemini）
  4. 回答會附上來源產品連結和相似度分數

何時需要重新索引？

目前需要手動 Re-index 當 Google Sheets 同步了新的產品資料（overview、features、specs 有變動）後，需要到 /ask 頁面點 Re-index 讓向量資料庫更新。

系統會自動比對 content hash，只重新 embed 有變動的 chunks，所以 re-index 很快且不浪費 API 費用。

未來可以整合到 Sync 流程中自動觸發。

9 Persona 系統

不同部門的同事對同一個問題需要不同角度的回答。Persona 機制讓你為每個使用情境定義獨立的 AI system prompt。

內建 Personas（4 個）

Persona適用對象回答風格
Product Specialist 預設 一般查詢、MKT 精確規格、型號比較、技術細節，中英文自動偵測
Sales Assistant 業務團隊 強調賣點、競爭優勢、使用場景、客戶導向語言
Technical Support 客服 / 技術支援 簡單語言、步驟式說明、相容性確認、troubleshooting
Product Manager PM 內部分析 詳細規格比較表、功能差距分析、跨產品線觀察

自訂 Persona

到 /settings/personas 可以：

  • 編輯內建 Persona — 修改 system prompt（可隨時 Reset 回預設值）
  • 新增自訂 Persona — 例如為合作夥伴、經銷商、特定市場建立專屬角色
  • 每個 Persona 有：ID（slug）、名稱、圖示、描述、System Prompt
自訂 Persona 範例

Partner Portal — 為合作夥伴設計：用英文回答、附上型號和訂購資訊、強調 partner program 相關內容。

Japan Market — 為日本市場設計：固定用日文回答、使用日本市場的產品命名、參考日文翻譯版本的內容。

API 整合（給不同部門做前端）

/api/ask 是獨立的 REST endpoint，任何前端都能呼叫。不同部門可以各自建介面，只要帶上不同的 persona 參數：

// 業務團隊的前端
POST /api/ask
{
  "question": "ECC100 跟海康威視比有什麼優勢？",
  "persona": "sales",
  "provider": "claude"
}

// 客服系統的 chatbot
POST /api/ask
{
  "question": "ECC100 支援哪些 PoE 標準？",
  "persona": "support",
  "provider": "gemini"
}

// 合作夥伴入口（自訂 persona）
POST /api/ask
{
  "question": "Which outdoor APs support WiFi 7?",
  "persona": "partner",
  "provider": "claude"

資料儲存

Persona 的 system prompt 存在 app_settings 表，key 格式為 persona_{id}。內建的 4 個 Persona 有 hardcoded 的預設值（在 personas.ts），DB 中的設定會覆蓋預設值。

10 各來源 Chunking 策略

Embedding 模型不變（都是 text-embedding-3-small），但不同來源需要不同的分割方式：

來源Chunking 策略每 Chunk 目標原因
product_spec 固定 2 chunks/product
(overview + specifications)		 200-1000 tokens 結構固定、資料量小，不需要複雜切割
gitbook 按 Markdown heading（h1/h2/h3）分割
長 section 用 recursive split		 300-800 tokens Markdown 結構清晰，heading 就是天然分割點
google_doc 按 heading 分割
無 heading 時用 recursive split		 300-800 tokens 格式不一致，需要同時支援有/無 heading 的文件
web 先用 Readability 提取正文
再按 <section> 或段落分割		 300-800 tokens HTML 結構雜，需要先去除 nav/footer/ads
file (PDF) 按頁面分割 + overlapping window
（50 tokens overlap）		 300-800 tokens 頁面跨段落是常見問題，overlap 確保不漏前後文
file (Word) 按 heading 或段落分割 300-800 tokens Word 的結構比 PDF 乾淨，可直接用段落
text_snippet 不切割，整段就是一個 chunk 自由 手動輸入的內容通常短小
Chunking 核心原則

每個 chunk 應該是「自含」的 — 單獨讀也能理解意思。

  • 切太小（一句話）→ 失去上下文，搜尋結果碎片化
  • 切太大（整份文件）→ 語意被稀釋，搜尋精準度降低
  • 目標：300-800 tokens / chunk，除非資料天然更小（如 text_snippet）

Recursive Splitting 策略

當一個 section 超過 800 tokens 時，需要進一步切割：

  1. 先按 \n\n（段落）分割
  2. 如果單段仍超過上限，按 \n（換行）分割
  3. 如果單行仍超過上限，按句號分割
  4. 相鄰 chunks 保留 50 tokens 的 overlap（確保前後文不斷裂）

不同來源的 metadata 建議

來源metadata 範例
product_spec{"product_line": "Cloud Camera", "status": "active", "chunk_type": "overview"}
gitbook{"collection": "Cloud Camera", "path": "setup/quick-start", "section": "Installation"}
web{"url": "https://...", "domain": "engenius.com", "page_title": "ECC100 Product Page"}
google_doc{"title": "Product Brief", "folder": "Marketing/Camera", "owner": "terrel@..."}
file{"filename": "ECC100_Datasheet.pdf", "file_type": "pdf", "pages": 3, "page_number": 2}
text_snippet{"author": "Terrel", "tags": ["FAQ", "ECC100"], "category": "competitive"}

11 圖片處理策略

未來的文件來源（Gitbook、Google Docs、PDF、網頁）都可能包含圖片。向量資料庫只能搜尋文字，因此需要將圖片「文字化」。

圖片描述化（Image-to-Text）

文件中的圖片 | v ┌─────────────────────────┐ │ Claude Vision API │ 多模態 LLM │ 圖片 → 結構化文字描述 │ 理解圖片內容 └────────────┬────────────┘ v ┌─────────────────────────┐ │ 描述文字嵌入 Chunk │ [Image: Front panel showing │ 成為可搜尋的內容 │ LED indicator and reset button] └────────────┬────────────┘ v ┌─────────────────────────┐ │ Embedding + 向量搜尋 │ 正常搜尋流程 │ 圖片描述也能被匹配到 │ "reset button 在哪裡？" └─────────────────────────┘

為什麼用 Claude Vision？

  • 專案已有 Anthropic API Key，不需額外設定
  • Claude Vision 對技術圖片（hardware diagram、network topology）描述品質好
  • 可以指定描述重點（例如：「描述這張硬體圖中的每個介面和按鈕」）

描述 Prompt 範例

// 給 Claude Vision 的指示
"Describe this image in detail for a product database.
Focus on:
- Hardware interfaces, ports, and buttons visible
- LED indicators and their labels
- Physical dimensions or mounting details if shown
- Network topology or connection diagram details
Write in plain text, suitable for text search."

圖文關聯（Image-Context Binding）

圖片描述不應該獨立存在，而是和它前後的段落文字綁在同一個 chunk 裡：

// chunk.content 範例
"Hardware Overview
The ECC100 features a compact dome design suitable for
outdoor installation with IP67 weatherproof rating.

[Image: Front panel showing LED indicator (power/status),
reset pinhole button, and mounting bracket. Rear panel
with RJ45 PoE port (802.3af) and microSD card slot
(supports up to 256GB).]

The device supports wall and ceiling mounting via the
included bracket."

// chunk.metadata 中存圖片 URL
{
  "images": [
    {
      "url": "https://xxx.supabase.co/storage/v1/.../ecc100_hw.png",
      "alt": "ECC100 hardware diagram",
      "position": "inline"
    }
  ]
}
優點：搜尋到段落時，圖片也跟著出來

當有人問「ECC100 的 microSD 插槽在哪裡？」，向量搜尋會匹配到這個 chunk（因為描述文字包含 "microSD card slot"），前端可以從 metadata 中取出圖片 URL 一起顯示。

層級儲存位置用途

實作時機與注意事項

目前不需要做

圖片處理只有在處理 Gitbook / Google Docs / PDF / 網頁 時才需要。目前 product_spec 的圖片（產品照、hardware 圖）不需要 embed — 它們是用來生成 PDF 的，不是用來回答問題的。

等到實作其他來源的 ingestion 時再加入圖片處理即可。documents 表的 metadata JSONB 欄位已經預留了存圖片資訊的空間，不需要改 schema。

注意事項說明
Vision API 費用 Claude Vision 每張圖約 $0.01-0.05（依解析度）。批量處理大量圖片前要估算費用
描述品質 技術圖表（hardware diagram）效果好；純照片（產品照）效果普通；文字截圖建議直接 OCR
圖片變更偵測 圖片沒有 content hash 機制。如果圖片更新了但 URL 不變，需要手動 force re-index
隱私 圖片會上傳到 Claude Vision API。確保不包含敏感資訊（內部設計稿、未發布產品等）

項目說明
OpenAI Key 是必要的 Embedding 固定使用 OpenAI text-embedding-3-small。即使用 Claude 回答問題，embedding 仍需 OpenAI key
索引需手動觸發 目前 Sync 後不會自動 re-index。同事更新了 Google Sheet → Sync 完成後，需要到 /ask 頁面點 Re-index
Content hash 防浪費 Re-index 時自動比對 SHA-256 hash，內容沒變的 chunk 不會呼叫 OpenAI API，不浪費費用
IVFFlat 索引的 lists 參數 目前設定 lists = 10，適合 < 1000 筆文件。如果未來文件數量超過 1000，需要增加 lists 數量或改用 HNSW 索引
相似度門檻 設定為 0.3（較寬鬆）。如果發現搜尋結果不相關太多，可以調高到 0.4-0.5
LLM 費用 每次提問約消耗 2000-4000 tokens（context + 回答）。以 Claude Sonnet 為例，約 $0.01-0.03 / 次。小團隊每月數百次查詢，費用可忽略
無 Auth 目前 /api/ask endpoint 沒有認證保護。任何知道 URL 的人都能使用。未來加入 Supabase Auth 時一併處理
回答品質取決於資料品質 如果 Google Sheet 的 overview/features 寫得不完整或不準確，AI 的回答也會受影響。「Garbage in, garbage out」
Persona prompt 長度 System prompt 不宜過長（建議 < 500 tokens）。太長會壓縮 context 的空間，影響回答品質
pgvector 在 Supabase Free Plan pgvector 包含在所有 Supabase 方案中（含免費方案）。不需要額外付費或開通

PDF/Word 是最複雜的來源

PDF 的格式問題很多：多欄排版會被解析成交錯文字、表格可能變成亂序、掃描檔需要 OCR、圖片中的文字無法直接提取。建議：

  • 先處理「文字型 PDF」（我們自己生成的 Datasheet PDF），這些格式規律
  • 掃描型 PDF 和複雜排版的第三方文件最後再做
  • 考慮使用更強的解析服務（如 Unstructured.io）取代本地 parsing

Text Snippets（手動文字片段）

項目說明
source_typetext_snippet
取得方式使用者在管理介面手動貼入文字
用途FAQ、內部公告、產品差異說明、競品比較、常見客戶問題的標準回答
Chunking不切割，整段就是一個 chunk
source_id使用者自訂的 slug
圖片處理不適用（純文字）
需要開發管理 UI（CRUD 頁面）+ 簡單的 ingestion
Text Snippet 是最快能產生價值的擴充

如果同事常常問同類問題（例如「ECC100 跟競品比怎麼樣？」），你可以直接貼入一段標準回答作為 text snippet，AI 就會在回答中引用這段內容。不需要任何外部 API 或解析邏輯。

另一個用途：把內部常用的「產品定位說明」「功能差異表」「市場策略文件」摘要貼入，讓業務團隊可以快速查詢。
