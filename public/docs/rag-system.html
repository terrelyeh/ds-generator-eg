Ask SpecHub

RAG 系統架構與使用說明 — 2026-04-13

1功能概述

Ask SpecHub 是內建於 Product SpecHub 的 AI 產品問答助理。同事可以用自然語言（中文或英文）詢問 EnGenius 產品相關問題，系統會從向量資料庫檢索相關資料，再由 LLM 生成帶來源引用的精準回答。

主要功能

功能範例
規格查詢「ESG510 的 VPN throughput 是多少？」
產品比較「ECC100 和 ECC500 差在哪裡？」
篩選搜尋「哪些 AP 支援 WiFi 7？」
產品推薦「推薦一台適合戶外的攝影機」
追問對話「那你可以幫我比較這幾台的規格嗎？」
多語言自動偵測問題語言，以相同語言回答

關鍵術語

術語說明
RAGRetrieval-Augmented Generation — 先搜尋再生成，LLM 基於檢索到的上下文回答
Embedding文字轉為 1536 維向量，語意相近的文字 = 距離相近的向量
pgvectorPostgreSQL 向量搜尋擴充，內建於 Supabase（含免費方案）
Chunk文件被切分為小片段，每片獨立做 Embedding 並可被搜尋
Persona系統提示詞角色，定義 AI 對不同受眾的回答風格

2系統架構

查詢流程 (Query Workflow)

使用者提問 (+ 對話歷史)
    → 結合歷史脈絡豐富查詢
    → Embed (OpenAI text-embedding-3-small)
    → 向量搜尋 (pgvector, top 8)
    → 載入 Persona 系統提示詞
    → 組建上下文 (歷史 + 匹配文件 + 問題)
    → LLM 回答 (Claude / GPT / Gemini — 使用者選擇模型)
    → 回應 + 來源引用
    → 自動儲存到 chat_sessions DB

索引流程 (Indexing Workflow)

Products DB → 建立 chunks (overview + specs per product)
    → Content hash 檢查 (跳過未變動的)
    → 批次 Embed (OpenAI)
    → Upsert 到 documents 表

3三維度 Prompt 架構

系統使用 3 層 Prompt 組裝來決定 AI 如何回答。每個維度獨立且可組合，形成一個回應風格的矩陣。

Prompt 組裝：
  Layer 1: Persona        (回答角度 — 從什麼角度回答)
  Layer 2: User Profile   (對話對象 — 誰在問)
  Layer 3: Output Format  (產出格式 — 要生成什麼)  [未來]
  Layer 4: RAG Context    (檢索到的產品資料)
  Layer 5: User Question  (實際問題)

3.1 維度一：回答角度 (Persona)

控制強調什麼內容、如何結構化回答，不只是語氣。

Persona強調重點「ECC100 PoE?」回答差異
Product Specialist精確規格、相關型號「802.3af，最大 15.4W」
Sales Assistant客戶價值、搭配建議「支援標準 PoE，一般企業 Switch 就能供電」
Technical Support實務檢查、故障排除「802.3af（15.4W）。確認 Switch 有開 PoE」
Product Manager跨型號模式、功能缺口「802.3af。ECC500 系列用 802.3at，這是分水嶺」
同一個問題，四種角度： 問「ECC100 PoE?」— Product Specialist 給精確規格數字、Sales 從客戶角度解釋供電相容性、Support 提醒實際檢查步驟、PM 分析產品線定位差異。4 個內建 Persona + 可在 /settings/personas 自訂。

3.2 維度二：對話對象 (User Profile)

根據提問者身份調整解釋深度與術語使用。

Profile行為調整
一般同事 (預設)不做額外調整 — 基線回應
新進同仁解釋術語、加入產品線背景、對基礎問題有耐心
業務人員實用答案，可直接面對客戶使用，建議搭配方案
Channel Sales市場定位、sell-through 策略、競爭優勢
產品經理完整技術深度、表格、功能差距分析
終端客戶最簡單語言、不用術語、聚焦好處而非規格
零摩擦設計： 預設「一般同事」不附加任何額外 Prompt。Profile Prompt 以獨立區段附加在 Persona Prompt 之後。每個 Session 會記住選擇，對話中不需重新選取。

3.3 維度三：產出格式 (Output Format)

控制生成什麼格式的內容，超越一般問答。未來

範本使用場景
Q&A (目前)問問題、得到回答
業務簡報為特定市場 + 產品生成 Pitch Deck 大綱
產品比較表結構化比較表
Email 草稿給客戶的產品推薦郵件
提案文件含痛點 + 解決方案的提案文件

Channel Sales × Education 市場：

維度 1：Sales Assistant（強調價值、建議搭配）
維度 2：Channel Sales（市場定位、sell-through 策略）
維度 3：業務簡報（結構化 Pitch 格式）未來

+ RAG：ECW536、ECW526、ECS5512FP 規格

→ AI 生成結構化 Pitch，包含市場痛點、產品推薦理由、競爭定位

3.5 外部 API 整合

/api/ask 端點是獨立的 REST API，外部系統可直接呼叫：

# 內部：業務團隊前端
POST /api/ask { "persona": "sales", "profile": "sales-rep", ... }

# 內部：客服系統 chatbot
POST /api/ask { "persona": "support", "profile": "customer", ... }

# 外部：合作夥伴入口
POST /api/ask { "persona": "partner", "profile": "channel-sales", ... }
未來：API Key 存取控制（Supabase Auth 之後）— 每個合作夥伴/部門取得唯一 API Key，可限制 source_type 搜尋範圍（如：夥伴只能搜 product_spec），並追蹤使用量（呼叫次數、Token 消耗）和依層級限速。

4資料庫設計

documents 表（通用，支援所有來源類型）

documents (
  id           UUID PRIMARY KEY,
  source_type  TEXT NOT NULL,       -- 'product_spec' | 'gitbook' | 'web' | 'google_doc' | 'file' | 'text_snippet'
  source_id    TEXT NOT NULL,       -- model_name, URL, Doc ID 等
  source_url   TEXT,                -- 用於回答中的引用
  title        TEXT NOT NULL,
  chunk_index  INT DEFAULT 0,
  content      TEXT NOT NULL,
  token_count  INT,
  metadata     JSONB DEFAULT '{}', -- product_line, locale, tags, images 等
  embedding    VECTOR(1536),       -- OpenAI text-embedding-3-small
  content_hash TEXT,                -- SHA-256，未變動則跳過 re-embed
  created_at   TIMESTAMPTZ,
  updated_at   TIMESTAMPTZ
)

chat_sessions 表（對話持久化）

chat_sessions (
  id            UUID PRIMARY KEY,
  user_id       TEXT DEFAULT 'anonymous', -- 未來：Supabase Auth user ID
  title         TEXT,                     -- 自動從第一則訊息生成
  persona       TEXT DEFAULT 'default',
  provider      TEXT DEFAULT 'gemini-flash',
  messages      JSONB DEFAULT '[]',       -- [{role, content, sources?, provider?}]
  message_count INT DEFAULT 0,
  created_at    TIMESTAMPTZ,
  updated_at    TIMESTAMPTZ
)

來源類型 (Source Types)

source_type狀態說明
product_spec已完成產品概述 + 規格，來自資料庫
text_snippet計畫中手動文字片段（FAQ、競品分析）
gitbook計畫中Gitbook 技術文件頁面
google_doc計畫中Google Docs（透過 Drive API）
web計畫中網頁內容
file計畫中上傳的 Word / PDF 檔案

5AI 模型與 Embedding

每次查詢涉及 兩次 AI 呼叫：

Step 1：Embedding（文字 → 向量）

項目內容
模型OpenAI text-embedding-3-small (1536 dims)
費用$0.02/1M tokens（約 $0.001 索引全部 66 個產品）
執行時機索引（每個 chunk）+ 每次使用者提問

Step 2：LLM 回答（上下文 + 問題 → 回答）

使用者可選擇 Provider → 再從下拉選單選特定模型（含等級 Badge）。

ProviderStrongestMainstreamBest CP
Gemini2.5 Pro2.5 Flash 預設2.0 Flash Lite
GPT4o4o Mini4.1 Nano
ClaudeOpus 4Sonnet 4Haiku 3.5
防幻覺機制： Prompt 指示「絕不捏造 context 中沒有的規格」。當問題模糊時，AI 會先追問釐清。語言自動偵測：問中文答中文、問英文答英文。

6Chunking 策略

每個產品生成 2 個 Chunks：

Chunk內容
Chunk 0 (Overview)型號名稱、副標、概述文字、重點功能
Chunk 1 (Specs)完整規格表，以 [Category] label: value 格式

目標：300-800 tokens/chunk。產品規格天然短小，不需要複雜的切分邏輯。

各來源 Chunking 策略（未來）

來源策略備註
product_spec固定 2 chunks/product已實作
gitbook依標題切分 (h1/h2/h3)Markdown 天然切分點
google_doc標題切分 + 遞迴處理有/無標題的文件
webReadability 萃取 + 段落切分先移除導覽列/頁尾/廣告
file (PDF)頁面切分 + 50 token 重疊處理跨頁段落
file (Word)標題/段落切分結構比 PDF 更乾淨
text_snippet不切分手動輸入，通常很短

功能說明

8對話歷史與持久化

運作方式

#說明
1前端每次 API 呼叫帶上最近 20 則訊息作為 history[]
2後端以歷史脈絡豐富 Embedding 查詢 — 讓「這幾台」能對應到上一輪提到的型號
3前輪對話納入 LLM Prompt 作為上下文
4若向量搜尋無結果但有歷史，LLM 從對話脈絡回答
5自動儲存：每次交換後，對話儲存到 chat_sessions 表（1 秒 debounce）
6Sidebar：漢堡圖示開啟歷史面板 — 點擊恢復、刪除或開新對話

Session 管理

項目說明
標題從第一則使用者訊息自動生成
Persona & Provider隨 Session 儲存，恢復時自動還原
user_id預設 anonymous — Supabase Auth 之後加 WHERE 條件即可
排序最近更新優先，顯示訊息數與相對時間

10設定與使用

首次設定檢查清單

  • 設定 OpenAI API Key（Embedding 必需），前往 /settings/api-keys
  • 設定至少一組 LLM Key（Claude / GPT / Gemini 擇一），用於回答
  • 確認 Migration 00009（documents）和 00010（chat_sessions）已套用
  • 前往 /knowledge → 點擊 Product Specs 卡片的 Index → 等待約 10-20 秒索引 66 個產品

日常使用步驟

步驟操作
1點擊 Navbar 的 Ask
2回答角度 — 選擇 Persona（Product Specialist / Sales / Support / PM）
3對話對象 — 選擇 User Profile（一般同事 / 新進同仁 / 業務 / Channel Sales / PM / 客戶）
4選擇 AI 模型（Gemini / GPT / Claude → 從下拉選單選特定模型）
5輸入問題或點擊範例
6追問更多問題（系統記住上下文）
7過去對話自動儲存，出現在 Sidebar（漢堡圖示）

12注意事項

項目細節
OpenAI Key 必須即使用 Claude/Gemini 回答，Embedding 仍需 OpenAI Key
Re-index 手動操作Sync 後需手動到 /knowledge 點擊 Re-index
Content hash避免 re-index 時不必要的 API 呼叫
IVFFlat lists=10適合 < 1000 筆文件，超過需調整
相似度閾值 0.3若結果太多不相關，調整為 0.4-0.5
無身份驗證/api/ask 目前未保護，廣泛推廣前需加 Supabase Auth
pgvector 免費Supabase 所有方案（含免費）皆包含
Persona Prompt 長度建議 < 500 tokens，留空間給 context
Markdown 渲染AI 回答以 Markdown 渲染（表格、列表、粗體、程式碼）via react-markdown + remark-gfm
對話自動儲存對話存入 DB，頁面重新整理不會遺失
模型選擇使用者先選 Provider → 再從帶等級 Badge 的下拉選單選模型
