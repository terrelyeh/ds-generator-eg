3Datasheet PDF 命名規則

3.1 子資料夾命名

每個型號在 DS 資料夾內有一個子資料夾：

{DS_Prefix}_{Model}/

範例:

產品線型號資料夾名稱
Cloud APECW526DS_Cloud_ECW526/
Cloud SwitchECS5512FPDS_Cloud_ECS5512FP/
Cloud CameraECC500DS_Cloud_ECC500/
Cloud VPN FWESG510DS_Cloud_ESG510/
Cloud NVSEVS1004DDS_Cloud_EVS1004D/
ExtenderEXT1106DS_Cloud_EXT1106/
Unmanaged SWES105DS_Unmanaged_ES105/
歷史格式: 部分舊資料夾名稱包含版本號（如 DS_Cloud_ECW526_v1.4/），系統可自動辨識。新建的資料夾一律不帶版本號。

3.2 PDF 檔案命名

{DS_Prefix}_{Model}_v{Major}.{Minor}.pdf

範例:

  • DS_Cloud_ECW526_v1.4.pdf
  • DS_Cloud_ECS5512FP_v1.5.pdf
  • DS_Unmanaged_ES105_v1.0.pdf

3.3 版本規則

規則說明
格式v{Major}.{Minor}  例: v1.4
新產品從 v1.0 開始
更新Minor +1  例: v1.4 → v1.5
版本來源系統自動掃描 DS 資料夾中的 PDF 檔名，取最新版本

版本偵測邏輯:

  1. 掃描型號對應的子資料夾
  2. 檢查資料夾名稱中的版本號（舊格式相容）
  3. 檢查資料夾內所有 PDF 檔名中的版本號
  4. 取最大版本號作為 base version
  5. 生成新 PDF 時自動 +1

4產品圖片命名規則 (DS Images)

4.1 檔案命名格式

{Model}_{type}.{ext}                      ← 英文版
{Model}_{type}_{locale}.{ext}             ← 多語言版（僅 hardware）
圖片類型type 名稱說明範例
產品照product產品正面照 (白底去背)，跨語言共用ECW526_product.png
硬體標示圖 (英文)hardware標示各介面/LED 位置ECW526_hardware.png
硬體標示圖 (日文)hardware_ja日文標註版本ECW526_hardware_ja.png
硬體標示圖 (繁中)hardware_zh繁中標註版本ECW526_hardware_zh.png
Radio PatternH-plane / E-planeAP 專用，跨語言共用ECW526_2.4G_H-plane.png

系統同時搜尋 .png 和 .jpg，優先使用 .png。

跨語言共用 vs 各語言獨立:
  • Product 圖和 Radio Pattern 只有一份（英文版），跨語言共用，放在英文產品線的 DS Images/ 內
  • Hardware 圖可以每個語言獨立（因為圖上的 LED / Port 標註文字不同），放在對應語言產品線的 DS Images/ 內

4.2 Radio Pattern (僅 AP 產品線)

{Model}_{Band}_{Plane}.png

範例:

  • ECW526_2.4G_H-plane.png / ECW526_2.4G_E-plane.png
  • ECW526_5G_H-plane.png / ECW526_5G_E-plane.png
  • ECW536_6G_H-plane.png / ECW536_6G_E-plane.png (Wi-Fi 6E)

檔名為什麼還要帶 _ja / _zh 後綴？
雖然語言版的 hardware 圖已經放在對應語言的 DS Images/ 裡，檔名後綴仍然保留是為了：
  1. 避免 Supabase Storage 路徑撞車（Storage 是 flat key，沒有 locale 目錄）
  2. 檔案被下載或轉寄時仍能自我識別
  3. 跟 PDF 檔名 DS_Cloud_ECW526_v1.0_ja.pdf 的命名慣例對齊
  4. 防呆：如果誤把 _ja 檔放到 _zh 資料夾，sync 可以偵測錯誤

7同步機制與圖片管理

7.1 真源 & 快取原則

核心原則: Google Drive 是所有資料的 authoritative source (真源)。Supabase 上的檔案（Storage）和 URL（DB 欄位）都是 快取副本，用於讓前端快速讀取。無論是 UI 上傳或 Drive 直接丟，Drive 那份永遠代表真實狀態；Supabase 會透過 Sync 自動對齊。
儲存位置角色誰寫入誰讀取
Google Drive DS Images/真源PM 手動 + Web UI write-through + Sync (建子資料夾)Sync 拉進 Supabase
Supabase Storage images/快取 (CDN)UI upload + Sync前端 preview / dashboard
Supabase DB products.*_imageURL 指標UI upload + Sync前端

7.2 同步排程

方式觸發時間說明
自動每天 09:00 (台灣時間)Vercel Cron 自動觸發，僅同步有變更的產品線
手動 (全部)隨時Dashboard 右上角「Sync」按鈕，強制同步所有產品線
手動 (單產品)隨時Product Detail 頁面 Resync Images 按鈕，只重新同步該產品的圖片（不碰 Sheets / 文字規格）

7.3 同步流程

  1. 讀取 Google Sheet (Web Overview + Detail Specs) → 更新產品資料 + 規格
  2. 掃描英文 {ProductLine}/DS Images/ → 同步 product / hardware 圖到 Supabase Storage，更新 products 表
  3. 對每個已啟用的語言，掃描 {ProductLine}_{locale}/DS Images/ → 同步 hardware 圖，更新 product_translations.hardware_image
  4. 掃描各語言的 DS 資料夾 → 偵測最新 Datasheet 版本號
  5. 比對現有資料 → 偵測變更 → 記錄 Change Log
  6. 有變更時 → 發送 Telegram 通知

7.4 刪除傳播

當 PM 從 Drive 刪除一張圖片（或拖進垃圾桶），系統會在下次同步時偵測並清除對應的 DB 欄位：

偵測方式行為
Sync 列出 DS Images 資料夾 (folder_listed = true)，但預期檔案不在清單中 清除 products.product_image / products.hardware_image / product_translations.hardware_image
Sync 列資料夾失敗（Drive API 錯誤）→ folder_listed = false 不做任何事（保守策略，防止網路毛刺導致誤刪）
注意: Sync 只清 DB 欄位（URL 指標），不會自動刪除 Supabase Storage 上的檔案本身。Storage 上的檔案會變成孤兒（不被引用），但不會顯示在前端。

7.5 UI 刪除功能

除了 Drive 端的自然刪除 + Sync 傳播外，MKT 可以直接在網頁介面刪除圖片：

位置操作影響
Product Detail → Product Images 各圖片右下角的紅色垃圾桶按鈕 同時刪除 Supabase Storage 檔案 + Drive 檔案 (trash) + 清 DB 欄位
Product Detail → Radio Pattern 各 slot 的垃圾桶按鈕 同上 + 刪除 image_assets 記錄
Translations → Hardware Image Delete 連結 刪除該語言版本的 hardware 圖（Supabase + Drive + product_translations.hardware_image）
Drive 刪除可復原: 所有透過 UI 執行的 Drive 刪除都是「移到垃圾桶」(trash)，不是永久刪除。30 天內可以從 Drive 垃圾桶中恢復。

8快速檢查清單

新增一個產品時，需要確認以下事項：

  • Google Sheet Web Overview 填寫完整 (Model Name, Model Number, Status, Headline, Single Overview, Key Feature Lists)
  • Google Sheet Detail Specs 填寫完整
  • 英文產品線 {ProductLine}/DS Images/ 放入 {Model}_product.png
  • 英文產品線 {ProductLine}/DS Images/ 放入 {Model}_hardware.png
  • (AP only) {ProductLine}/DS Images/ 放入 Radio Pattern 圖片
  • (多語言) 對應語言產品線 {ProductLine}_{locale}/DS Images/ 放入 {Model}_hardware_{locale}.png
  • 確認 Status 欄位正確 (Active / Upcoming / Pending)
  • 執行一次 Sync 確認資料正確匯入

9.6 Hardware 圖片多語言

Hardware Overview 頁面的圖片可以依語言不同使用不同版本（因為 LED / Port 標註文字不同）。產品照和 Radio Pattern 永遠跨語言共用。

        └── ECC500Z_hardware_zh.png    ← 繁中版
自動建立子資料夾: 若 Cloud Camera_ja/ 下面還沒有 DS Images/ 子資料夾，系統首次上傳或同步時會自動建立。但上層的 Cloud Camera_ja/ 必須事先存在（這是 PM 決定新增語言支援時手動建立的）。

{Model}_hardware_{locale}.{ext}
  • 日文: ECC500Z_hardware_ja.png
  • 繁中: ECC500Z_hardware_zh.png（zh-TW locale 在檔名中縮寫為 zh）

9.6.4 前端顯示邏輯

  • Preview 頁面讀取 product_translations.hardware_image（per-locale）
  • 該語言有上傳 → 使用語言版本
  • 該語言未上傳 → fallback 使用英文版的 products.hardware_image
真源 & 快取: Google Drive 是圖片的 authoritative source，Supabase Storage 是快取（給前端快速讀取）。無論是 UI 上傳或 Drive 直接丟，Drive 那份永遠是真實狀態；Supabase 會被 Sync Cron 自動對齊（包含刪除）。

9.7 翻譯工作流程

  1. 進入產品的 Translations tab
  2. 點 + Enable Language 啟用語言（如 Japanese）→ 狀態為 Draft
  3. 翻譯 Product Headline、Overview、Features（手動輸入或 AI 翻譯）
  4. (選) 上傳語言專屬的 Hardware Image
  5. (選) 設定 QR Code 標籤和連結
  6. 點 Preview 確認排版效果
  7. 點 Save & Confirm → 狀態變為 Confirmed
  8. 從 🌐 按鈕 Generate PDF
重要: 必須 Save & Confirm 之後才能 Generate PDF。Preview 可以隨時使用但不算正式確認。

9.8 Spec Label 翻譯（Full 模式）

Spec Label 翻譯是 per-product-line（產品線層級），同一產品線的所有型號共用同一套翻譯。

  • 入口：Dashboard toolbar → Translations 連結
  • 每個產品線獨立一個管理頁面
  • 包含 Section Header 翻譯 + Spec Label 翻譯
  • 修改會影響該產品線所有型號的 Full 模式 Datasheet

9.9 AI 翻譯 & 詞庫

系統內建 AI 翻譯功能，支援多家 AI 模型：

Provider模型需要 API Key
AnthropicClaude Sonnet 4.6 / Opus 4.6ANTHROPIC_API_KEY
OpenAIGPT-4oOPENAI_API_KEY
Google AIGemini 2.5 ProGOOGLE_AI_API_KEY

API Key 可以在 Settings 頁面設定（存到 DB），或透過 Vercel 環境變數設定。Settings 優先於 env var。

AI 翻譯特點:

  • 翻譯同時改善原文品質（修正不通順的句子結構、統一術語）
  • 5 層 Prompt 系統：基礎翻譯規則 → 語言規則 → 產品線術語 → 內容類型規則 → 公司詞庫
  • 翻譯完成後顯示翻譯筆記（用繁體中文說明做了哪些優化）
  • AI Model 下拉只顯示已設定 API Key 的模型，未設定的會反灰

翻譯詞庫 (Glossary):

  • 在 Settings → Manage Glossary 管理公司認可的翻譯術語
  • 詞庫分 Global（全公司通用）和產品線專屬（如只套用於 Cloud Camera）
  • AI 翻譯時自動載入詞庫，確保術語一致性
  • 範例：Night Vision → ナイトビジョン (JA) / 夜視 (ZH)

9.10 CJK 排版

日文 / 中文版 Datasheet 自動套用以下排版規則：

  • 禁則處理 (line-break: strict) — 句號、逗號不出現在行首
  • 兩端對齊 (text-align: justify) — 右邊不參差不齊
  • CJK 字型 — 日文使用 Noto Sans JP，繁中使用 Noto Sans TC
  • 行高和字級針對 CJK 文字微調，確保可讀性
